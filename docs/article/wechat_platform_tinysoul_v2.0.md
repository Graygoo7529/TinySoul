# TinySoul：从一次 Query 开始，理解 Agent 的运行内核

我在做一个小型 Agent 框架，叫 TinySoul。它现在还不是一个完整的 Agent 平台，也不是一个可视化工作流工具。它更像一个原型实验：先把 Agent 推理和执行的每一步骤做清楚。

> 当用户提出一个问题或任务后，Agent 如何围绕这一次请求，经过多轮判断、执行、记录和修正，最后稳定地给出结果？

这里我用 Query 来指代“一次用户请求”。它可以是一个简单问题，也可以是一件需要多步完成的任务，比如读取文件、分析内容、生成报告，再给出总结。

TinySoul 的目标并不是一开始就做大，而是先回答一个更具体的问题：如果我们不把 Agent 只看成一段提示词，而是看成一个有状态、有动作、有记录、能处理错误的小系统，那么它的内核应该怎么设计？

---

## 为什么先从一次 Query 做起

很多 Agent 项目一开始会自然走向“大系统”：多用户、多会话、长期记忆、插件市场、任务队列、权限管理、可视化编排。

这些方向都重要，但如果底层的一次 Agent 运行还没有讲清楚，系统很快会变成提示词、工具调用、状态更新和异常处理混在一起的复杂物。它也许能跑通几个 demo，但当结果不稳定时，很难回答几个关键问题：

- 模型为什么选择了这个动作？
- 动作执行后的结果被谁使用了？
- 状态为什么变成了现在这样？
- 失败以后应该终止，还是把错误反馈给模型继续修正？
- 如果状态更新失败，刚才的执行结果会不会丢失？

TinySoul 选择先收窄范围，从单个 Query 开始。

这里的“单个 Query”不是“一问一答”。它可以包含多轮动作，也可以中途向用户追问信息，还可以读写工作区文件、调用模型、执行脚本、生成中间结果。它的关键在于：一次任务有一个明确目标，有一条可以观察和复盘的运行轨迹。

这种收窄有两个好处。

第一，它让调试变得具体。我们可以看到 Agent 每一轮为什么这么做、做了什么、结果是什么、下一步状态如何变化。

第二，它为更复杂的 Agent 打基础。长期来看，一个通用 Agent 不应该把完整聊天记录一股脑塞进上下文，然后期待模型自己消化一切。更合理的做法，是围绕当前 Query 主动收集必要的上下文、文件资源、长期记忆、可用工具和技能，再在这一次任务里规划和使用它们。

所以 TinySoul 的基本立场是：先把 Query 作为 Agent 行为的基本组织单位。

## 我做了什么：一个可运行的 Agent 内核原型

TinySoul 当前已经实现了一个比较完整的原型骨架。它围绕一次 Query，提供了几类核心能力。

第一，是固定的运行循环。

Agent 每一轮不是自由地“想一想然后随便调用工具”，而是按固定步骤推进：先决定下一步要做什么，再为这个动作准备参数并执行，最后根据执行结果更新任务状态。

第二，是可描述、可执行的动作系统。

每个动作都需要说明自己能做什么、适合什么时候用、需要哪些输入、执行后可能带来什么影响。模型根据这些说明选择动作，框架负责真正执行。

第三，是显式的任务状态。

TinySoul 会记录当前还有哪些待办事项、已经完成了哪些关键进展、每次动作执行产生了什么结果、有哪些错误需要继续处理。这样 Agent 的进展不会只藏在自然语言对话里。

第四，是独立的工作区。

如果任务涉及文件，TinySoul 会把“任务进展”和“文件世界”分开管理。任务状态记录 Agent 理解到哪里了，工作区记录有哪些文件、资源和变更。

第五，是统一的错误和控制流。

动作成功、失败、超时、用户中断、后台任务返回进度，这些情况不会散落在各处各自处理，而是进入统一的信号和错误处理路径。

第六，是一些实验性能力。

比如动态动作：模型可以在受控范围内创建一个临时脚本，再把它注册成新的动作供后续步骤调用。再比如持续运行的后台动作：它不阻塞主流程，而是通过后续信号汇报进展。

这些能力还没有到生产可用的程度，但它们已经能帮助我验证一个核心判断：Agent 不应该只是“模型加工具列表”，而应该有一套清楚的运行协议。

---

## 为什么把一次运行拆成三步

TinySoul 的主循环里，一轮运行被拆成三步。

第一步，选择动作。

模型看到当前任务、已有进展、待办事项、工作区概况，以及可用动作的简要说明。它只需要回答一个问题：下一步应该做什么？

第二步，生成参数并执行动作。

模型只看到刚才选中的那个动作的详细说明，然后生成这个动作需要的参数。真正的执行由框架完成。

第三步，更新状态。

动作执行结束后，系统会产生执行记录。模型阅读这些新记录，再决定是否新增待办、完成待办、取消待办，或者写入一个新的关键进展。

这样拆分看起来比“让模型自己决定并调用工具”更麻烦，但它带来了边界。

选择动作时，模型不需要关心参数细节。生成参数时，模型不需要重新比较所有工具。更新状态时，模型不直接操作文件，也不直接执行外部命令。

也就是说，模型仍然负责判断，但判断发生在一个明确的协议里；框架负责执行、记录、路由和恢复。

我希望 TinySoul 形成的是这样的 Agent 内核：不是让模型完全自由发挥，而是让模型在一个可观察、可调试、可恢复的小系统里工作。

---

## 为什么动作要“自我说明”

在很多实现里，工具就是一个函数，再加一段给模型看的说明。

TinySoul 对动作的要求更高一点：每个动作都要能描述自己。它需要说明自己适合做什么、不适合做什么、需要什么输入、执行后可能影响什么。

这里又做了一层拆分。

模型在选择动作时，只看到简要说明。它只需要知道“这个动作大概能做什么、什么时候该用”。

当动作被选中后，模型才会看到更详细的参数说明和示例，用来生成具体输入。

这样做的原因很朴素：如果每一轮都把所有工具的完整参数说明塞给模型，上下文会很快变长，而且大量无关细节会干扰判断。真实的人类工作也不是这样。我们通常先决定要用哪个工具，再去看这个工具怎么用。

把“选择工具”和“使用工具”分开后，动作目录更容易扩展，提示词也更容易维护。

同时，动作的“说明”和“执行”也要分开。模型只需要理解动作能力，框架负责管理超时、取消、记录、依赖和运行环境。这样动作既能被模型理解，也能被工程系统稳定管理。

---

## 为什么执行动作不只是函数调用

如果 Agent 只调用一些很快返回的小函数，问题确实不复杂。

但真实任务里，动作可能要读写文件、调用模型、运行脚本、访问外部命令，甚至启动一个持续运行的后台任务。它可能成功，也可能失败；可能很快完成，也可能超时；可能只有一个，也可能多个并发执行。

所以 TinySoul 把动作执行看成一个独立的调度问题。

即使当前只执行一个动作，它也会走统一的执行路径。将来多个动作并发执行时，不需要再维护另一套完全不同的逻辑。

超时也被分成不同层次。一次动作本身有生命周期预算，动作内部如果调用模型也有请求预算，并发执行时还有整体收口的时间边界。这样可以避免一个慢动作拖住整个循环，也可以避免模型请求长时间挂起。

后台动作则采用另一种方式：启动成功后先返回，后续进展通过信号通知系统。停止后台动作时，也不是靠名字猜，而是通过一次执行的唯一标识来追踪。

这些设计听起来偏工程，但它们解决的是 Agent 系统里很实际的问题：动作不只是函数调用列表，而是一组需要被调度、记录、超时控制和恢复处理的执行单元。

---

## 为什么要显式管理状态

Agent 的状态很容易被藏在自然语言历史里。

比如模型说：“我已经读完文件，下一步要生成报告。”这当然表达了一种状态，但系统很难稳定地根据这句话判断哪些事情真的完成了、哪些还没完成、哪些错误需要下一轮继续处理。

TinySoul 选择把状态显式化。

它会记录待办事项、关键进展、动作执行记录，以及需要反馈给模型的错误。每一轮运行时，这些状态都会重新进入上下文，模型不是凭感觉继续，而是在明确状态上继续。

同时，TinySoul 把任务状态和工作区分开。

任务状态回答的是：Agent 认为这件事进行到哪里了？

工作区回答的是：外部文件世界里现在有什么资源，发生过哪些变更？

这两件事不能混在一起。更新任务进展不应该直接改文件；读写文件也不应该只靠一句自然语言描述。文件操作应该通过明确动作完成，并留下执行记录和资源变更记录。

这种分离让系统更干净，也让复盘更容易。

---

## 为什么长文本不应该直接进入状态

在 Agent 系统里，还有一个很容易被忽略的问题：到底什么内容应该进入状态，什么内容应该留在文件世界里？

如果每次动作执行后，都把完整结果塞进状态，系统很快会变得臃肿。比如读取了一篇长文、生成了一份报告、分析了一批数据，如果这些长文本都进入运行状态，后续每一轮都会带着越来越大的上下文继续运行。

这会带来两个问题。

一是成本会不断上升。任务越长，模型要读的内容越多，其中很多内容对当前步骤并不重要。

二是语义会变得混乱。状态本来应该回答“任务进行到哪里了”，而不是承担文件存储的职责。把长文本、文件内容、执行日志都混进状态里，后面很难判断哪些是任务进展，哪些只是外部资源。

所以 TinySoul 做了一个边界划分：动作执行后，只把简短结果、摘要和必要引用写入状态；完整内容尽量放在 Workspace 里。

举个例子，一个动作读取了一份文档，它不需要把整篇文档都写入状态，而是返回“已读取某个文件、主要内容是什么、文件路径是什么”。如果后续动作确实需要这份文档，它可以在参数里明确指定要引用哪个文件，框架再临时把对应内容加载进来。

这是一种渐进式披露：模型平时只看到资源列表、摘要和变更记录；只有在某一步真正需要文件内容时，才加载具体文本。

这样做的好处是，State 保持轻量，负责记录任务进展、待办、关键结果和错误；Workspace 负责承载文件、长文本和资源变化。两者并列存在，但职责不同。

这也是 TinySoul 对上下文管理的一个基本判断：Agent 不应该把所有内容都长期带在身上，而应该知道有哪些资源、资源在哪里，以及在需要时如何取用。

## 为什么模型调用也要分层

TinySoul 不希望业务代码到处直接调用模型。

一次 Agent 运行里，不同阶段对模型能力的要求并不一样。选择下一步动作，需要的是稳定判断；生成动作参数，需要的是遵循说明和结构化输出；更新状态，需要理解执行记录并产出清晰的状态变化；动作内部有时还需要单独调用模型完成生成或总结。

如果所有模型调用都走同一套配置，要么成本过高，要么能力不匹配。

所以 TinySoul 把不同阶段的模型调用分成不同配置。每类调用可以使用不同的模型链、超时、重试和生成参数。这样模型调用成为基础设施的一部分，而不是散落在各个动作里的临时请求。

这也让后续优化更直接。某一步不稳定时，可以单独调整这一步的模型、提示词和解析逻辑，而不必牵动整个系统。

---

## 为什么提示词也要单独管理

很多 Agent 项目的提示词最后会变成一个越来越大的字符串。

系统规则、任务说明、工具说明、用户偏好、历史记录、示例、错误反馈，都被塞进同一个上下文里。短期看很方便，长期看很难维护。

TinySoul 把提示词分成两类。

一类是稳定规则，比如 Agent 应该遵守的运行协议、每一步应该输出什么格式、动作执行时应该遵守什么边界。

另一类是动态上下文，比如当前 Query 是什么、现在有哪些状态、这一步新产生了哪些执行记录、当前工作区有什么资源。

这样拆分后，稳定规则可以长期维护，动态上下文可以围绕当前任务按需组装。

更长远看，这也为 Agent Home 留出了空间。Agent 的身份、团队规则、领域规范、提示词版本、可复用技能和资源索引，不应该永远硬编码在代码里，也不应该每次全部塞进上下文。

我希望未来 TinySoul 能围绕当前 Query 渐进加载真正需要的规则、记忆、工具和技能，而不是让上下文无限膨胀。

---

## 当前已经能做什么

到目前为止，TinySoul 已经具备一个 Agent 内核原型所需要的主要骨架：

- 可以围绕一次 Query 进行多轮运行。
- 每轮按“选择动作、执行动作、更新状态”推进。
- 动作可以自描述，并由框架统一执行。
- 可以管理待办、关键进展、执行记录和错误反馈。
- 可以读写工作区文件，并记录资源变化。
- 可以通过统一路径处理成功、失败、超时、中断和后台进度。
- 可以按不同阶段配置模型调用。
- 可以管理系统提示词和任务提示词。
- 已经有动态脚本动作和后台动作的实验性原型。
- 项目中也包含配置、日志、测试和模块文档。

但我仍然把它定义为原型。

这里的“原型”不是说代码随便，而是说它当前更适合研究、验证和个人实验。它已经可以帮助我验证 Agent 内核的结构，但还没有完成生产系统所需的持久化、权限、安全隔离、服务化和完整观测。

---

## 当前边界

TinySoul 现在有一些明确边界，需要讲清楚。

第一，长任务下的上下文裁剪还不够精细。框架已经有按需组装上下文的基础，但不同步骤目前还没有完全接入精细裁剪，任务变长后 token 成本仍然会增长。

第二，动作输入校验仍然偏轻量。当前主要校验输入是不是对象、必要字段是否存在，完整的类型、枚举、范围等校验还需要继续补齐。

第三，脚本沙箱不是生产级安全隔离。现在的限制适合实验，不适合直接执行不可信代码。真正生产使用还需要容器、低权限进程、系统调用限制等更强措施。

第四，高风险动作还需要框架级确认。涉及删除、覆盖、外部修改的动作，不能只依赖模型自己判断，应该由框架提供更明确的确认机制。

第五，状态还没有持久化层。当前一次任务的状态主要绑定在一次运行生命周期里。要支持中断恢复、多会话、长期记忆，还需要新的存储设计。

这些边界决定了 TinySoul 当前适合被怎么使用：它适合作为 Agent 架构原型、学习项目和个人研究框架，不适合直接当作生产级自动化平台。

---

## 后续规划

短期我会继续打磨单个 Query 内核。

动态脚本也会继续加强。重点包括失败回滚、沙箱边界、注册过程的可观察性，以及更明确的安全提示。

后台动作也会继续完善。现在它已经能启动、汇报进度和停止，但还需要更完整的列表、暂停、恢复和清理能力。

中长期，我会把重点放在 query-centered context，也就是“围绕当前 Query 组织上下文”。

我希望 Agent Home 能逐步承担更多职责：管理 Agent 的身份、规则、提示词版本、领域技能和资源索引。我也希望引入与单次运行状态分离的记忆和会话层。这样系统可以围绕当前 Query 检索长期记忆，而不是简单继承全部历史对话。

再往后，可以探索更结构化的语义地图，把“模型曾经说过什么、文件里有什么、任务需要什么”变成可检索、可压缩、可更新的语义资源。

TinySoul 的长期目标不是做一个更大的聊天窗口，而是做一个围绕 Query 组织上下文、工具、记忆和技能的 Agent 内核。

---

## 结语

TinySoul 现在还小，但它试图认真回答一个基础问题：

> 如果 Agent 不是一段神秘提示词，而是一个有状态、有动作、有记录、有错误处理的小系统，它应该如何运行？

我目前的答案是：从一次 Query 开始，用固定运行循环约束多轮行为，用自描述动作管理能力，用显式状态记录任务进展，用工作区管理外部资源，用统一的执行记录和信号机制处理成功、失败和恢复。

这些设计不是为了把系统做复杂，而是为了让 Agent 的运行过程变得可观察、可调试、可扩展，并且在失败后知道如何继续。这还不是最终形态，但它已经是一个可以继续演进的内核。
